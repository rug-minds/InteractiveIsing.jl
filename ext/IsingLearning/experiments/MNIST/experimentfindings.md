# MNIST Experiment Findings

Use of this file: essential findings needed to make MNIST experiments run correctly. Keep this short; do not archive every failed run here.

- The manager-backed MNIST path should reuse worker contexts. Rebuilding graphs/processes per batch makes epochs far too slow.
- Static model data should be shared across workers where possible. Sharing adjacency and base parameter arrays was the key scaling improvement; verify this before long runs.
- Worker state, clamping buffers, gradient buffers, and per-sample input fields must stay worker-local. Shared parameter arrays should be read during a minibatch and updated once after the flush.
- With only copied worker graphs/adjacency, scaling was much worse than expected. Sharing static arrays brought the synthetic/manager tests much closer to useful thread scaling.
- On this 16-core/32-logical-thread machine, `-t 32` with 32 workers is worth testing; 16 workers is not automatically better just because there are 16 physical cores.
- Batchsize equal to worker count is not a useful learning setup by itself. Workers need to accumulate their own buffers over multiple examples, and only then should buffers be synchronized.
- A batch is not an epoch. Compare timing by minibatch latency and by full selected-dataset epoch time separately.
- The interactive MNIST demo is for runtime/debug visualization. It should not use the training manager and should not change the graph state when loading images.
- MNIST image labels looked wrong once because image orientation/display was flipped, not because the MNIST labels were inherently wrong. Be careful to distinguish display orientation from dataset target encoding.
- Fixed input spins and image-dependent magnetic fields are physically equivalent ways to provide input. The field-based path is preferred for manager training because it avoids source graph state writes and works with shared static data.
- The paper-style small architecture is the first reliable baseline to keep using for diagnostics: `784 -> about 120 hidden -> 40 output`.
- The `40` output spins mean four replicas per digit; reduce them to ten class scores by averaging or summing each digit's four replicas.
- A short paper-style radius-7 manager diagnostic did learn on a small train/test subset, reaching roughly 0.89 test accuracy in 20 epochs. Use that as the sanity check before larger local-architecture grids.
- The earlier `10x` hidden-layer MNIST runs were useful for manager/scaling tests but overshoot the paper-sized architecture; do not use them as the first learning baseline.
- Local-neighborhood MNIST grids should not be launched until a short diagnostic shows accuracy improving and epoch time is sane. If it is flat, stop and inspect relaxation, input mode, output replicas, and gradient signs before widening the grid.
- Learning runs need enough relaxation sweeps. Raw step counts that are not full sweeps caused misleading non-learning diagnostics.
- Adam is the current practical optimizer default. Muon was discussed, but the current sparse/vector parameterization does not make it a drop-in optimizer.
- The old local paper-style file applied updates and clipping after each sample. The clean manager file accumulates minibatch gradients and updates once per flush, so old per-sample learning rates are not equivalent. With `gradient_normalization = :sum` the old LR scale collapsed to one class; with `:mean` it was stable but much weaker.
- For the clean two-hidden-layer CNN-style paper-manager `28^2 -> 11^2 -> 40` run, `batchsize = 32`, `gradient_normalization = :mean`, and 16x the old per-sample LR scale recovered useful learning. A 30-epoch `100/class` run reached best `20/class` test accuracy `0.755`; the best checkpoint evaluated at `0.719` on `100/class` with 3 reads/50 sweeps and `0.801` with 10 reads/75 sweeps. The 32x LR scale collapsed to single-class predictions.
- For the larger CNN-style `28^2 -> 14^2 -> 40` second hidden layer, the same 16x LR scale was too unstable/weak. Radius 5 with an 8x LR scale reached best `20/class` test accuracy `0.80`; the best checkpoint evaluated at `0.791` on `100/class` with 3 reads/50 sweeps, `0.854` with 10 reads/75 sweeps, and `0.861` with 30 reads/75 sweeps.
- In the current CNN-style radius checks, radius 5 is the best local fanout. For `11x11` hidden2, radius 7 reached `0.69` and radius 3 reached `0.385`. For `14x14` hidden2 at LR16, radius 3/5/7 were weak; radius 5 became useful after lowering the LR scale to 8x.
- The inlaid-input diagnostic uses a 55x55 input layer where the 784 MNIST pixel sites are fixed and the 2241 separator sites remain live. A static active-index set is enough; no source changes or whole-layer toggles are needed.
- For that inlaid-input diagnostic, 100 full Metropolis sweeps kept pixel states fixed and gave useful 32-worker scaling: with 8 jobs per worker, throughput was about `1008` jobs/s at 32 workers versus `65.5` jobs/s at 1 worker, roughly `15.4x`.
- The same inlaid-input relaxation probe still lowered energy noticeably from 50 to 75 sweeps and was close to flat by 100 sweeps, so 75-100 full sweeps is the first sensible range for training diagnostics.
- The first working inlaid-input trainer needed four fixes: use `[0, 1]` fixed pixel values rather than bipolar `[-1, 1]` pixels, separate target-field clipping from trainable bias clipping, negate the paper-style contrastive update before passing it to Adam, and add fixed output competition between different digit groups.
- The first saved inlaid-input run is `runs/current/inlaid_pixelreadout_beta20_comp05_lr001_e30_100pc`: `55x55` partially dynamic input, 40 output replicas, 32 workers, batchsize 256, 100 train examples/class, 40 test examples/class, 75/75 free/nudged sweeps, Adam `lr=0.01`, β=20, output competition 0.5, and pixel-only readout. It reached `0.61` test accuracy at epoch 30; post-hoc best-checkpoint eval also reached `0.61` with 3 reads/100 sweeps and 5 reads/150 sweeps.
- A stock `mnist_local_nn_grid.jl` diagnostic with `28x28 -> 28x28 -> 11x11 -> 40`, radius 5, 32 workers, batchsize 256, and LocalLangevin/quadratic clamping was stopped after a 5-minute timeout. It had only completed one epoch and stayed near chance, so the paper-style nudge/update remains the practical MNIST path for now.
- Track and save train/test accuracy, train/test loss, epoch time, best parameters, and final parameters for every run that learns.
